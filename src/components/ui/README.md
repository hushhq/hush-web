# `src/components/ui` — Import Convention

Two UI primitive families coexist in this directory:

1. **Legacy Hush primitives** — hand-written `.jsx` components built on
   Radix Themes / Radix Primitives, styled by the project's own CSS
   classes in `src/styles/global.css` (`.btn`, `.btn-primary`, etc.).
   - File names use **PascalCase**: `Button.jsx`, `IconButton.jsx`,
     `AlertDialog.jsx`, `Dialog.jsx`, `DropdownMenu.jsx`,
     `ContextMenu.jsx`, `Tooltip.jsx`, `ScrollArea.jsx`, `Select.jsx`,
     `Tabs.jsx`, `Switch.jsx`, `Separator.jsx`.
   - All exported through the aggregate `./index.js`.

2. **shadcn/ui primitives** — generated by `npx shadcn add <name>`.
   Tailwind-styled, registered via `components.json`.
   - File names use **lower-case** (matches the shadcn CLI default):
     `button.tsx`, future `card.tsx`, `dialog.tsx`, etc.
   - Imported **explicitly** by file path. They are intentionally **not**
     re-exported from `./index.js`.

## Import rules

| Source                | Import form                                                     |
|-|-|
| Legacy aggregate      | `import { Button, IconButton } from '@/components/ui'`          |
| Legacy named file     | `import { DialogRoot } from '@/components/ui/Dialog'`           |
| shadcn (explicit)     | `import { Button as ShadcnButton } from '@/components/ui/button'` |

Why explicit shadcn imports:

- `Button` is the most common identifier collision. Re-exporting the
  shadcn `Button` from `./index.js` would silently shadow the legacy
  `Button` for every consumer that imports from `@/components/ui`,
  forcing a global migration on day one. Keeping shadcn explicit means
  consumers opt in per-call-site, with the alias making the intent
  obvious in code review.
- The case difference (`Button.jsx` vs `button.tsx`) plus
  `forceConsistentCasingInFileNames: true` in `tsconfig.json` makes
  each path unambiguous: `'./Button'` always resolves to the legacy
  component, `'./button'` always to the shadcn one.

## Migration policy

- New product UI may use **either** family. Pick one per surface — do
  not mix legacy and shadcn primitives in the same component without a
  reason.
- When a legacy primitive is fully superseded by its shadcn equivalent,
  migrate consumers in a dedicated PR:
  1. Switch consumers to `@/components/ui/<name>`.
  2. Run the test + build suite.
  3. Delete the legacy file and its `index.js` export in the same
     commit.
- Until that point, legacy imports must keep working unchanged.
